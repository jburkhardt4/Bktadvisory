# BKT Advisory — Agent Second Brain

This file is the **Context Engineering** layer for this repository. It provides condensed, always-available domain knowledge and project rules so that any AI agent working here can retrieve the right context immediately — without overloading on irrelevant detail.

---

## The Five Pillars of Agentic Engineering

### 1. Context Engineering
Context is best served fresh and condensed. Provide the exact right amount of information — not the maximum. Store domain knowledge, product specs, and business rules directly in the codebase as markdown so agents can retrieve them easily. This file is that "second brain."

### 2. Agentic Validation
Give agents a way to self-validate their own work. Build automated validation loops — browser screenshots, ADB simulators for mobile, structured logs — so quality assurance does not depend on a human reviewing every change.

### 3. Agentic Tooling
Identify and remove friction that slows down an agent's execution loop. Whenever an agent gets stuck and requires manual human intervention, build a CLI tool, skill, or system specifically for the agent to use so that humans are no longer the bottleneck.

### 4. Agentic Codebases
Optimize the codebase for the next AI agent, not the next human engineer. Because agents are probabilistic, dead code and competing framework patterns can poison context. Actively clean up bad patterns, keep file structures consistent, add agent-specific logging, and encode golden principles directly into the repository.

### 5. Compound Engineering
When the entire team buys into agentic engineering, the knowledge and workflows compound over time. As team members continuously add new tools, scripts, and domain context into the shared codebase, agents eventually self-validate and largely build the software autonomously.

## Golden Rules for This Repository

### 1. Project Boundary
This repository is the **marketing frontend and client portal for bktadvisory.com**. No unrelated business logic belongs here.

### 2. Tech Stack
**Vite + React + Tailwind v4.0 + TypeScript.** Do not introduce alternative build tools, UI frameworks, or CSS methodologies.

### 3. Strict Icon Rule
**Do NOT use `lucide-react`.** All icons must be built as inline SVG components. If a component currently imports from `lucide-react`, it must be refactored to use an inline SVG instead.

### 4. Figma Integration
All UI changes must reference the connected **'BKT Advisory' Figma file** for design tokens, spacing, typography, and visual specifications. Never guess at design values — pull them from Figma.

### 5. Validation Rule
Before finishing any UI task, you **must** take a screenshot using the browser tool and compare it to the Figma design tokens. This is a non-negotiable self-validation step.
